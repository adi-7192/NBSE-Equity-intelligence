# Phase 02.1: Workspace Shell & Personal Integrations - Context

**Gathered:** 2026-03-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the product shell so the application feels like a coherent private terminal instead of a single dashboard page. This phase covers the reusable app shell, primary navigation, dark-theme visual direction, better typography and spacing, a dedicated settings area, and secure per-user integration settings for Zerodha and supported AI providers. It does not yet expand screener, stock-research, portfolio analytics, or macro/news scope beyond making those destinations navigable placeholders if needed.

</domain>

<decisions>
## Implementation Decisions

### Workspace shell
- **D-01:** The app should move to a reusable shell with a left sidebar for primary navigation and a slim top bar for workspace status, search, and user actions.
- **D-02:** Primary sidebar items for day one should be Dashboard, Archives, Screener, Portfolio, Settings, and Admin only for admin users.
- **D-03:** The shell should be designed to scale into later phases instead of feeling like a one-page dashboard retrofit.

### Visual direction
- **D-04:** The product should stay dark-themed, but with a stronger terminal-finance identity rather than a generic dashboard look.
- **D-05:** The core palette should use a charcoal/graphite base with a warm amber accent.
- **D-06:** Red and green should be reserved primarily for market semantics, not used as general accent colors.
- **D-07:** The interface should become denser and more information-rich, but with cleaner spacing, better alignment, and a stronger typography hierarchy.

### Settings information architecture
- **D-08:** Settings must be a dedicated full page, not a small modal or an afterthought section on the dashboard.
- **D-09:** Settings should use a secondary left sub-navigation so it can scale cleanly.
- **D-10:** Top-level settings sections should be Profile, Integrations, Security, and Workspace Status.

### Personal integrations model
- **D-11:** Zerodha credentials should be stored per user as personal settings, not as workspace-wide admin settings.
- **D-12:** AI provider credentials should also be stored per user as personal settings.
- **D-13:** First-class AI providers in this phase are OpenAI, Gemini, and Anthropic.
- **D-14:** Users should only be asked for the credentials that are durably required to make the connection work; the app should absorb unnecessary protocol complexity.

### Security and credential handling
- **D-15:** Secrets must be stored safely server-side with proper protection and never exposed raw in the frontend.
- **D-16:** Saved credentials should be masked in the UI after entry and represented through status metadata such as configured, last updated, or validation state.
- **D-17:** The frontend may show connection health and setup guidance, but secret values themselves must never be retrievable in plaintext from the browser.

### Integration UX behavior
- **D-18:** Zerodha should feel like a user-friendly connection flow, not a manual token-pasting workflow.
- **D-19:** When a user has not configured Zerodha or AI yet, the dashboard should still load normally and show polished empty states rather than broken widgets.
- **D-20:** Settings should be the main place to complete setup, and the app should only hard-block when a user attempts a feature that truly requires a missing integration.

### the agent's Discretion
- Exact sidebar component implementation and responsive collapse behavior.
- Exact font pairing and spacing scale, so long as it supports the denser dark-terminal visual direction.
- Exact connection-status copy, empty-state illustrations, and validation micro-interactions.
- Exact secure-storage implementation details, as long as secrets remain encrypted, server-side, and non-recoverable from the frontend.

</decisions>

<specifics>
## Specific Ideas

- The current dashboard should feel more like a private market terminal and less like a demo page with tabs.
- The shell should make future areas like Screener and Portfolio feel like first-class destinations even before those phases are fully built.
- Zerodha connection UX should prioritize simplicity: ask for the minimum durable credentials and let the app handle the rest.
- AI integration setup should feel consistent across OpenAI, Gemini, and Anthropic rather than three unrelated forms.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `components/ui/*`: existing shadcn/Radix primitives can support navigation, settings forms, and status cards without introducing a new UI library.
- `app/layout.tsx`: existing root layout is the right integration point for a reusable shell wrapper.
- `app/page.tsx` and `components/dashboard-client.tsx`: current dashboard surface shows the information hierarchy problems this phase needs to solve, but also provides reusable live-market and weekly-intelligence sections.
- `app/admin/page.tsx`: existing admin route can be folded into the shell and role-aware sidebar rather than remaining an isolated page.
- `lib/auth.ts` and middleware/auth guards: existing user/admin separation can drive role-aware navigation and settings access.

### Established Patterns
- Protected routes and user-aware pages are already server-gated, which supports a shell that can vary by role and integration state.
- Tailwind + shadcn + Radix are the established presentation stack, so the redesign should stay within those patterns.
- The product already distinguishes dashboard data, archives, and admin surfaces; the issue is organization and UX, not a lack of route-level separation.

### Integration Points
- The new shell should become the persistent frame around Dashboard, Archives, future Screener and Portfolio pages, Settings, and Admin.
- Personal integration settings will need new secure server-side storage tied to the authenticated user model from Phase 1.
- Live-market and AI features should consume per-user integration status from settings-aware server boundaries rather than from global environment-only assumptions.

</code_context>

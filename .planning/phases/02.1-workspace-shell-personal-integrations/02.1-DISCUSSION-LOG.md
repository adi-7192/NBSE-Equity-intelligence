# Phase 02.1: Workspace Shell & Personal Integrations - Discussion Log

> **Audit trail only.** Do not use as input to planning, research, or execution agents.
> Decisions are captured in CONTEXT.md — this log preserves the alternatives considered.

**Date:** 2026-03-31
**Phase:** 02.1 Workspace Shell & Personal Integrations
**Areas discussed:** Navigation shell, settings structure, personal integrations, security posture, visual direction, missing-integration behavior

---

## Credential ownership

| Option | Description | Selected |
|--------|-------------|----------|
| Workspace-wide admin-managed credentials | One set of provider credentials shared across the workspace | |
| Per-user personal settings | Each user manages their own Zerodha and AI provider connections | ✓ |

**User's choice:** Per-user personal settings for both Zerodha and AI providers
**Notes:** Chosen to keep each user's connections independent and to support a real settings experience rather than a hidden operator-only config layer.

---

## Secret storage model

| Option | Description | Selected |
|--------|-------------|----------|
| Env-var references only | Settings page points to values managed outside the app | |
| In-app secure storage | Credentials are saved securely server-side and never exposed raw in the frontend | ✓ |

**User's choice:** In-app secure storage with proper protection
**Notes:** The user explicitly wants credentials safely stored with protection and not exposed in the frontend.

---

## Zerodha connection UX

| Option | Description | Selected |
|--------|-------------|----------|
| Manual technical flow | Ask the user for all token/exchange artifacts directly | |
| User-friendly minimal-input flow | Ask only for the durable credentials actually required and let the app handle the rest | ✓ |

**User's choice:** User-friendly minimal-input flow
**Notes:** The user wants the app to ask only for what is required to make Zerodha work for the product, not to expose technical token plumbing unless a later advanced mode is needed.

---

## AI provider scope

| Option | Description | Selected |
|--------|-------------|----------|
| One provider first | Start with a single AI provider | |
| Three first-class providers | Support OpenAI, Gemini, and Anthropic in the personal settings UX | ✓ |

**User's choice:** OpenAI, Gemini, and Anthropic
**Notes:** These should be first-class in the settings model from the start of this phase.

---

## Primary navigation

| Option | Description | Selected |
|--------|-------------|----------|
| Top-nav-first layout | Use a header bar as the primary app navigation | |
| Left sidebar + slim top bar | Use a scalable sidebar for destinations and a smaller top bar for utility/status | ✓ |

**User's choice:** Left sidebar plus slim top bar
**Notes:** Chosen because the product is growing into a multi-surface terminal and needs navigation that will scale past a single page.

---

## Sidebar contents

| Option | Description | Selected |
|--------|-------------|----------|
| Minimal shell | Only expose current implemented pages | |
| Future-ready shell | Show Dashboard, Archives, Screener, Portfolio, Settings, and role-aware Admin | ✓ |

**User's choice:** Future-ready shell
**Notes:** The shell should make future destinations feel intentional even before all of them are fully implemented.

---

## Settings information architecture

| Option | Description | Selected |
|--------|-------------|----------|
| One long stacked settings page | Keep all settings on a single scroll-heavy page | |
| Dedicated page with secondary left sub-navigation | Structure settings into Profile, Integrations, Security, and Workspace Status | ✓ |

**User's choice:** Dedicated page with secondary left sub-navigation
**Notes:** Chosen because the settings area needs to scale and should not feel bolted onto the dashboard.

---

## Visual direction

| Option | Description | Selected |
|--------|-------------|----------|
| Generic SaaS dashboard dark mode | Conventional dark UI with broad accent use | |
| Terminal-finance dark theme | Charcoal/graphite base, warm amber accent, market colors reserved for data semantics | ✓ |

**User's choice:** Terminal-finance dark theme
**Notes:** The user wants a better-designed dark UI with stronger typography, better alignment, and a more intentional terminal-like feel.

---

## Missing-integration behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Hard block the app until providers are connected | Prevent normal use until integrations are fully configured | |
| Graceful empty states with setup guidance | Let the dashboard load and guide users to Settings where needed | ✓ |

**User's choice:** Graceful empty states with setup guidance
**Notes:** The app should only hard-block when a user tries a feature that truly requires the missing integration.

## Deferred Ideas

- Advanced manual token/debug mode for Zerodha if the user-friendly flow proves insufficient
- Broader collaboration or workspace-level shared integrations
- Full feature implementation for Screener and Portfolio beyond navigation/readiness

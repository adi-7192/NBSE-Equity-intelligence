<user_constraints>
## User Constraints (from CONTEXT.md)

### Locked Decisions
- The app should move to a reusable shell with a left sidebar for primary navigation and a slim top bar for workspace status, search, and user actions.
- Primary sidebar items for day one should be Dashboard, Archives, Screener, Portfolio, Settings, and Admin only for admin users.
- The shell should be designed to scale into later phases instead of feeling like a one-page dashboard retrofit.
- The product should stay dark-themed, but with a stronger terminal-finance identity rather than a generic dashboard look.
- The core palette should use a charcoal/graphite base with a warm amber accent.
- Red and green should be reserved primarily for market semantics, not used as general accent colors.
- The interface should become denser and more information-rich, but with cleaner spacing, better alignment, and a stronger typography hierarchy.
- Settings must be a dedicated full page, not a small modal or an afterthought section on the dashboard.
- Settings should use a secondary left sub-navigation so it can scale cleanly.
- Top-level settings sections should be Profile, Integrations, Security, and Workspace Status.
- Zerodha credentials should be stored per user as personal settings, not as workspace-wide admin settings.
- AI provider credentials should also be stored per user as personal settings.
- First-class AI providers in this phase are OpenAI, Gemini, and Anthropic.
- Users should only be asked for the credentials that are durably required to make the connection work; the app should absorb unnecessary protocol complexity.
- Secrets must be stored safely server-side with proper protection and never exposed raw in the frontend.
- Saved credentials should be masked in the UI after entry and represented through status metadata such as configured, last updated, or validation state.
- The frontend may show connection health and setup guidance, but secret values themselves must never be retrievable in plaintext from the browser.
- Zerodha should feel like a user-friendly connection flow, not a manual token-pasting workflow.
- When a user has not configured Zerodha or AI yet, the dashboard should still load normally and show polished empty states rather than broken widgets.
- Settings should be the main place to complete setup, and the app should only hard-block when a user attempts a feature that truly requires a missing integration.

### the agent's Discretion
- Exact sidebar component implementation and responsive collapse behavior.
- Exact font pairing and spacing scale, so long as it supports the denser dark-terminal visual direction.
- Exact connection-status copy, empty-state illustrations, and validation micro-interactions.
- Exact secure-storage implementation details, as long as secrets remain encrypted, server-side, and non-recoverable from the frontend.

### Deferred Ideas (OUT OF SCOPE)
- Full screener feature implementation remains Phase 3.
- Full portfolio analytics and holdings workflows remain Phase 5.
- Broader visual redesign of stock detail and research surfaces belongs with Phase 4.
- Any collaboration, shared team workspaces, or multi-user workspace administration beyond current admin controls remains out of scope.
</user_constraints>

<research_summary>
## Summary

This phase needs one coherent approach rather than separate “UI cleanup” and “settings feature” work. The standard expert move in a Next.js app like this is to build a persistent authenticated shell at the layout layer, move page-specific content into route-local panels, and treat settings/integrations as first-class domain data with their own schema and server-only query/actions layer.

For the UI side, the repo already has the right primitives: Tailwind, shadcn/ui, Radix, `sidebar.tsx`, `sheet.tsx`, cards, separators, and scroll areas. That means the best path is not a new design library but a more opinionated composition system: a real app shell, denser typography, consistent section framing, and better empty/error/loading states. For personal integrations, the correct model is encrypted-at-rest storage in Postgres, with only masked metadata sent back to the client. The browser should submit credentials but never receive the stored secret back.

**Primary recommendation:** Split implementation into two plans: first ship the app shell and route structure, then add encrypted personal integration storage plus settings UX on top of that shell.
</research_summary>

<architecture_patterns>
## Architecture Patterns

### Recommended Project Structure
```text
app/
├── (app)/                 # authenticated shell routes
│   ├── layout.tsx         # workspace shell
│   ├── page.tsx           # dashboard
│   ├── archives/page.tsx
│   ├── settings/page.tsx
│   ├── screener/page.tsx  # placeholder for now
│   ├── portfolio/page.tsx # placeholder for now
│   └── admin/page.tsx
├── login/page.tsx         # shell-excluded auth entry
└── api/...
lib/
├── security/              # encryption and masking helpers
├── db/queries/            # integration query/actions
└── integrations/          # provider-specific mapping/status helpers
```

### Pattern 1: Authenticated Route Group Shell
**What:** Move protected product pages under a shared route-group layout so sidebar, top bar, and workspace framing are defined once.
**When to use:** Whenever multiple pages share navigation, role-aware actions, and status UI.
**Why here:** The repo currently duplicates navigation/session controls per page, which is exactly what the route-group shell solves.

### Pattern 2: Secret Envelope + Metadata Split
**What:** Store encrypted secret payload separately from user-facing metadata such as provider, configured status, last updated, validation state, and label.
**When to use:** Any time a user needs to manage secrets in-app without ever retrieving them raw.
**Why here:** Settings pages need to show meaningful connection state without turning the DB into plaintext credential storage or the client into a secret reader.

### Pattern 3: Graceful Capability Gating
**What:** Let the shell and pages render normally, but gate only the parts that need an integration behind setup cards and empty-state messaging.
**When to use:** Multi-surface products where a missing provider should not collapse the whole workspace.
**Why here:** The user explicitly wants a friendly product, not an integration wall at login.

### Anti-Patterns to Avoid
- **Per-page nav duplication:** creates drift and guarantees more UI cleanup later.
- **Plaintext secrets in DB or props:** unacceptable for a personal integrations feature.
- **Dump-everything settings page:** fast to build, but it becomes unreadable as more providers and security options arrive.
- **Using green/red as generic theme accents:** clashes with market semantics and hurts trust in price/status signals.
</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Shell built only for today’s routes
**What goes wrong:** The layout looks good for Dashboard and Archives but collapses once Screener, Portfolio, and Settings deepen.
**Why it happens:** Designers optimize around the current page count instead of the milestone map.
**How to avoid:** Build around the roadmap destinations now, even if some routes are placeholders.
**Warning signs:** Nav labels or active states feel like exceptions rather than a stable app frame.

### Pitfall 2: Secret handling stops at “not shown in UI”
**What goes wrong:** Credentials are hidden visually but still stored plaintext or sent back through API payloads.
**Why it happens:** Teams conflate “masked on screen” with “secured.”
**How to avoid:** Encrypt before persistence, keep decryption server-only, and separate metadata from secret material.
**Warning signs:** The client can fetch previously entered values, or the DB schema stores provider secrets in obvious text columns.

### Pitfall 3: Dark mode without hierarchy
**What goes wrong:** Everything is dark, but nothing is readable or aligned; cards blur together and typography feels mushy.
**Why it happens:** Dark-theme work focuses on colors only, not structure.
**How to avoid:** Treat hierarchy, spacing, and type rhythm as the primary design contract; color is secondary.
**Warning signs:** Too many similar dark surfaces, weak headings, and no clear primary/secondary reading path.
</common_pitfalls>
